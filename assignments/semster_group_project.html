Web Programming

Fall 2014 Semester Group Project

Overview

You will be working on a team of 4 or 5 people (see below) to create a prototype pertaining to personal health and fitness. Please note that a prototype is different than a product: a product is a complete system with documentation while a prototype simulates most of the complete system. I do not expect a full-blown product to be completed by the end of the semester.

Goal of Semester Group Project

Have students work in a team to define a problem, build and demo a solution.

Unacceptable Project Ideas

  • A blog
  • A message board

Security and Privacy of Data

Because you will be potentially working with health information, you need to carefully consider the security and privacy of using and collecting health and personal data. Please read:

  • Mobile Health and Fitness Apps: What Are the Privacy Risks? (privacyrights.org)
  • As Apple Moves into Health Apps, What Happens to Privacy? (WSJ)
  • App developers seek clearer U.S. health data privacy guidelines (Reuters)

Absolute Requirements

Your team's project must adhere to the following non-negotiable requirements:

  1. Must use HTML5.
  2. Must use JavaScript, jQuery, and Ajax.
  3. The team must use GitHub for the project. Each team will be provided a private repo in GitHub under the tuftsdev organization.
  4. Must have desktop and mobile web user interfaces. That is, the user interface is different on the desktop than on mobile.
  5. Must use a server-side framework or technology (e.g., Node.js, Django, Ruby on Rails)
  6. Must use least 2 web / RESTful APIs (e.g., RunKeeper, MyFitnessPal, Google Fit) with JSON and/or XML parsing

Pick Five

On top of the absolute requirements, the project must use at least 5 of the following technologies or ideas (your team's choice):

  • Geolocation
  • Server-side data persistence (e.g., with PostgreSQL, MongoDB, MySQL)
  • Client-side data persistence with either local storage or web SQL
  • Front-end framework including Bootstrap, React, Backbone.js, AngularJS
  • Data or screen scraping
  • Reporting with charts and graphs
  • Send emails, SMSes, or push notifications

Assessment

Your team will be graded on five factors:

  1. Functionality - Something that works consistently. Remember, a project with fewer features that work is significantly better than one with numerous features, in which some either do not work or are buggy.
  2. Utility - Does your project fulfill a recognized need? Is it also easy to understand?
  3. Originality - C'mon, another running app or calorie counter? Better if your team strives to do something more cosmic.
  4. Technical Competency - Use good programming practices, consistent Git commits and pushes, meaningful Git commit messages used. Egregious data collecting will be penalized.
  5. Teamwork - Does all team members contribute? Having one or two people doing all the work is not good. On the last day of class, Thursday, December 4th, every person will conduct a performance review on each teammate!
